# AI 产品经理面试评估框架

基于 AI 转型能力模型 v1（4+1 能力结构），采用四象限分类法。

---

## 评估总原则

**上下文闭环**：好的回答必须有三层——**因（为什么）→ 做（怎么执行）→ 果（结果+沉淀）**。缺层就是不完整。

---

## 通用失分筛子

以下四类问题在所有象限中通用，发现即标记：

| 类型 | 表现 |
|---|---|
| 具体性缺失 | 满嘴大词，没有数字、案例、决策节点 |
| 逻辑链断裂 | 跳步、因果倒置、取巧回避 |
| 自洽性存疑 | 和简历、前后回答相互矛盾 |
| 深度不扛压 | 浅层复述，追问一层就崩 |

## 象限一：自我介绍与转型动机、岗位选择类

**它回答什么：** 我是谁 / 我为什么转 / 我为什么适合

**对应能力：** 能力5「求职转译与专业表达」

**企业判断信号：**
- 能否把自己的项目经历整理成清晰案例，体现业务、产品、AI 和落地判断
- 能否在简历中准确表达自己的角色、动作、判断和结果，而不是堆概念
- 能否在面试中讲清楚问题背景、决策过程、方案取舍和最终产出
- 能否用作品、文档、原型或 Demo 等材料呈现自己的能力证据
- 能否根据不同岗位要求调整表达重点，让企业快速判断匹配度

**拆分逻辑：** 案例组织 → 简历表达 → 面试讲述 → 作品证明 → 岗位匹配

### 评估矩阵

| 评估维度 | 优秀 | 存疑 |
|---|---|---|
| **叙事线** | 有自己的版本，不是简历流水账；有因果，不是罗列 | 按时间线平铺，讲完跟没讲一样 |
| **自我定位** | 亮点有棱角，知道自己独特在哪 | 亮点全员通用（"执行力强、善于沟通"） |
| **转岗逻辑** | 能说清离开理由+看重方向，两者逻辑一致；对目标岗位有具体认知 | 动机飘忽、纯抱怨前公司/领导、把"想挣钱"包装成"追求成长" |
| **岗位匹配** | 知道自己和这个岗位的关系是什么，能一句话说清匹配逻辑 | 说不清自己经历和AI产品经理的关系 |

### 一句话压测

> 他是**想清楚了**转岗，还是**干不下去了**转岗？

## 象限二：产品能力与项目落地路径类

**它回答什么：** 我是否具备产品基本功 / 能不能把项目做成

**对应能力：** 能力1「业务场景理解与问题定义」+ 能力2「产品结构设计与交互表达」+ 能力4「方案整合与落地验证」

### 能力1：业务场景理解与问题定义

**企业判断信号：**
- 能否识别清楚目标用户、关键角色，以及多方角色之间的关系
- 能否还原真实业务场景，讲清楚当前事情是如何发生和运转的
- 能否定位关键问题，区分表面需求、真实问题和主要矛盾
- 能否判断问题价值，说明这个问题为什么值得解决、优先级为什么高
- 能否识别业务约束，讲清楚资源、流程、组织、数据或合规等限制条件

**拆分逻辑：** 角色对象 → 业务场景 → 关键问题 → 问题价值 → 现实约束

### 能力2：产品结构设计与交互表达

**企业判断信号：**
- 能否定义清楚产品中的核心对象、信息结构和字段关系
- 能否设计清楚页面、模块和功能边界，而不是只描述抽象想法
- 能否说明用户输入、系统处理和结果输出之间的关系
- 能否讲清楚用户操作路径、系统反馈、状态变化和异常情况
- 能否让产品结构具备基本可实现性，让人感觉方案可以被理解、评审和实现

**拆分逻辑：** 信息对象 → 页面模块 → 输入输出 → 交互状态 → 可实现性

### 能力4：方案整合与落地验证

**企业判断信号：**
- 能否把业务问题、产品结构和 AI 判断整合成一套逻辑一致的完整方案
- 能否明确方案范围，区分 MVP、后续迭代和暂不处理的内容
- 能否设计清楚落地路径，说明先做什么、后做什么、依赖什么资源
- 能否定义验证指标，说明如何判断方案是否真的有效
- 能否设计兜底机制，说明方案失败、AI 出错或效果不达预期时如何处理

**拆分逻辑：** 方案完整性 → 范围切分 → 落地路径 → 效果验证 → 失败兜底

### 评估矩阵

| 评估维度 | 对应能力 | 优秀 | 存疑 |
|---|---|---|---|
| **用户理解** | 能力1 | 能说清产品给谁用、不同角色需求的冲突 | 说不清自己做的产品到底给谁用 |
| **场景还原** | 能力1 | 能还原真实业务流，不是只讲功能列表 | 只会讲功能，不会讲业务为什么这么跑 |
| **问题定义** | 能力1 | 能判断"什么问题值得解决"，先定义再动手 | 拿到需求直接做，不问为什么 |
| **约束识别** | 能力1 | 方案里有现实感，不是纸上谈兵 | 不考虑现实约束 |
| **产品结构** | 能力2 | 能说清对象建模，有页面/模块设计思维 | 只会说"做一个XX功能"，说不出具体结构 |
| **交互状态** | 能力2 | 正向流程+异常边界都想到 | 只想正向流程，不考虑异常和边界 |
| **可实现性** | 能力2 | 方案被开发评审时不抓瞎 | 方案天马行空，开发看完想打人 |
| **方案闭环** | 能力4 | 方案逻辑一致，拼图能拼起来 | 方案碎片化，各模块对不上 |
| **范围切分** | 能力4 | 知道先做哪个后做哪个，不是大而全 | 贪大求全，说不清重点 |
| **落地路径** | 能力4 | 有清晰的执行路径和资源意识 | 只有目标没有步骤 |
| **效果验证** | 能力4 | 能说清怎么算成功，不是上线即结束 | 说不出衡量标准 |
| **兜底设计** | 能力4 | 有Plan B意识 | 没想过失败，只想成功路径 |
| **ownership** | — | 结果归因合理，主动提失败/遗憾 | "我们"分不清，成绩归自己、失败甩锅 |

### 一句话压测

> 他是项目的**主演**，还是**观众**？

## 象限三：AI能力与技术判断类

**它回答什么：** 我是否理解 AI 在真实业务中的接入方式和能力边界

**对应能力：** 能力3「AI能力理解与技术判断」

**企业判断信号：**
- 能否判断某个业务节点是否适合使用 AI，并说明适合或不适合的原因
- 能否选择相对合理的 AI 接入方式，例如大模型、RAG、工作流、Agent、工具调用或传统规则
- 能否设计清楚人机分工，说明哪些环节由 AI 完成，哪些环节需要人工判断、审核或接管
- 能否定义 AI 输出质量的判断方式，包括准确性、稳定性、可控性和可评估性
- 能否识别 AI 方案的关键约束，包括成本、延迟、数据安全、幻觉风险和业务风险

**拆分逻辑：** 适配判断 → 技术路径 → 人机分工 → 质量评估 → 风险约束

### 评估矩阵

| 评估维度 | 优秀 | 存疑 |
|---|---|---|
| **适配判断** | 能判断"用不用"比"怎么用"更重要；知道什么不该用AI | 不判断适不适合，上来就套AI |
| **技术路径** | 能说出不同路径的适用场景和取舍原因 | 只知道"用AI"，不知道用什么形态的AI |
| **人机分工** | 人机边界清晰，有具体的接管/审核节点 | 人机边界模糊，说不出人工介入时点 |
| **质量评估** | 有质量标准，知道怎么算"够好" | 没有质量标准，不知道怎么评估AI产出 |
| **风险约束** | 有多维度trade-off视角，方案里风险和收益并存 | 只说优点不说局限 |

### 一句话压测

> 他是在**合理用AI**，还是在**外包思考给AI**？

## 象限四：AI发散认知与成长潜力类

**它回答什么：** 我是否有持续学习能力、趋势判断和长期成长空间

**对应能力：** 能力3（认知层延伸）+ 持续学习能力

### 评估矩阵

| 评估维度 | 优秀 | 存疑 |
|---|---|---|
| **持续关注** | 能说出最近三个月关注的具体动态，不只是大模型的版本号 | 答不出最近关注过什么，或者只有一年前的旧信息 |
| **独立观点** | 有观点+有支撑逻辑，能说出**为什么**这么看 | 观点跟随大流，和别人说的完全一样；只能说好不能说局限 |
| **业务连接** | 能把一项新技术讲清楚"对我所在业务的影响是什么" | 趋势和业务两张皮，说得漂亮但落不了地 |
| **成长潜力** | 有主动学习路径（课程/项目/实践），不是被动等培训 | 说不出自己最近学了什么新东西 |

### 一句话压测

> 他对AI有**自己的观点**，还是**转述媒体**？

---

## 综合评级建议

| 评级 | 标准 |
|---|---|
| 🥇 强烈推荐 | Q2、Q3均在优秀区间，Q1无存疑 |
| 🥈 建议通过 | 仅一个象限存疑且可补救 |
| 🥉 谨慎通过 | Q2或Q3存疑（核心能力区失分），或两个象限存疑 |
| ❌ 不推荐 | Q1动机存疑（前置筛选不过）；或自洽性明显有问题 |
